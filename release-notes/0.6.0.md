# StepEngine 0.6.0

Release date: 2026-04-20

## Highlights

- Added a new `stepengine-reactor` module for Reactor-based workflows.
- Added a new `stepengine-core` module to hold shared retry, timeout, and failure contracts.
- Preserved the existing imperative `stepengine` module as the primary in-process engine for synchronous workflows.
- Improved test naming and coverage around exception propagation and compensation semantics.

## New modules

### `stepengine-reactor`

This release introduces a Reactor-native engine with the same core semantics as the imperative engine:

- sequential and parallel execution
- retry via shared `RetryPolicy`
- `ServiceException` as a non-retryable business failure
- per-step timeout with `StepTimeoutException`
- compensation in reverse order for completed sequential steps
- parallel-group compensation and wait-all failure aggregation

The reactive API includes:

- `ReactiveStep`
- `ReactiveParallelGroup`
- `ReactiveStepEngine`

### `stepengine-core`

Shared contracts have been extracted into `stepengine-core` so both engines depend on the same core types without depending on each other.

This module now owns:

- `CompensateOnError`
- `ServiceException`
- `StepTimeoutException`
- `RetryPolicy`
- `NoRetryPolicy`
- `ExponentialBackoffRetryPolicy`

## Documentation

- Updated the README for the multi-module layout.
- Added dependency guidance for `stepengine`, `stepengine-reactor`, and `stepengine-core`.
- Added a Reactor usage example and clarified imperative vs reactive timeout behavior.

## Testing

Release validation was run with:

```bash
mvn test
```

Results at release time:

- `stepengine-core`: 9 tests passed
- `stepengine`: 52 tests passed
- `stepengine-reactor`: 52 tests passed

## Upgrade notes

- Imperative users should continue depending on `stepengine`.
- Reactor users should depend on `stepengine-reactor`.
- Consumers that only need shared retry or exception contracts can depend on `stepengine-core`.
